How I Designed, Deployed, and Measured My Portfolio as a DevOps Platform

Last updated — March 2026

TL;DR — I treated this portfolio as a small production platform: opinionated architecture, automated CI/CD with environment separation, DORA-style delivery metrics, and explicit trade-offs. The goal is evidence — not screenshots.

  • Scope: Personal portfolio CI/CD environment
  • Data: Generated from real pipeline executions
  • Demo: Delivery Metrics Dashboard

Why I treated my portfolio like a production system

Delivery practices are best evaluated through artifacts. Treating the site as a platform creates traceable evidence: build outputs, gated previews, and measurable signals.

DevOps Decision
Prioritize reproducible pipelines and observable delivery over feature breadth.

Architecture overview

The system follows a split-plane architecture: a static presentation layer served via CDN, separated from an instrumented runtime surface that ingests CI events.

This separation limits blast radius, reduces cost, and allows meaningful instrumentation without over-engineering the marketing surface.

CI/CD design choices

Pipelines are split into preview and production stages. Preview builds validate changes with ephemeral credentials, while production deploys publish immutable artifacts.

CI/CD
Secrets are sourced from a secure store (vault-backed or equivalent, depending on environment), and runner images are pinned for reproducibility.

DORA metrics integration

Metrics include deployment frequency, lead time, change failure rate, and time to restore. Events are emitted directly from CI pipelines and aggregated for clarity.

The dashboard below is a live portfolio demo. If it does not load, the metrics are described in the sections above.

Delivery metrics dashboard (portfolio environment)

Representative, explainable metrics were prioritized over high-volume telemetry to maintain signal quality.

Monitoring and observability

Lightweight RUM sampling, CI event ingestion, and aggregated client-side errors are used. All data is scrubbed to avoid PII and limit storage cost.

Trade-offs and limitations

Static hosting simplifies security and cost but shifts runtime needs to isolated services. This trade-off keeps the platform maintainable for a single engineer.

Trade-off
Favor clarity of evidence over breadth of instrumentation; aggregation may hide rare edge cases.

What broke — and how I fixed it

  1. Over-instrumentation: noisy metrics and privacy risk → reduced sampling and aggregation.
  2. Preview drift: environment mismatch → rely on build-time inputs and mocked secrets.

Closing thoughts

This portfolio is treated as a platform: measurable, opinionated, and explicit about trade-offs. It converts claims into artifacts that hiring teams can validate quickly.
